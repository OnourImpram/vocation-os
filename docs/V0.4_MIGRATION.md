# v0.4 Local Runtime Migration

Version 0.4.0 moves consequential local runtime state into the encrypted SQLite event store owned by `vocationd`.

## Safety Properties

1. Planning is read only and deterministic.
2. The plan output contains hashes and opaque identifiers, not imported payload values.
3. Apply requires the exact reviewed `planHash`.
4. Legacy state, config, and ledger files remain unchanged after import.
5. Repeating the same import is idempotent.
6. An encrypted rollback backup is created before import events are written.
7. Existing encrypted stores authenticate and create a standard encrypted rollback backup before any migration writes.
8. Migration history and the exact v0.3 table and index contract are verified before adoption.

## Native Credential Route

Build the package and start the daemon:

```bash
npm run build
vocationd start
```

The native route uses Windows Credential Manager, macOS Keychain, or Linux Secret Service through `@napi-rs/keyring`. If the native provider is unavailable, startup fails without a shell or plaintext fallback.

## Headless Credential Route

Start both daemon and CLI commands with `--headless`:

```bash
vocationd start --headless
vocation daemon-status --headless
```

The master passphrase is read only from a masked interactive terminal. It is not accepted in an argument or environment variable. The resulting credential vault is encrypted with AES 256 GCM using a key derived by `scrypt`.

## Import Procedure

1. Start `vocationd`.
2. Run `vocation legacy-import-plan`.
3. Review `valid`, warnings, errors, source counts, and candidate hashes.
4. Resolve every error. Unknown but valid JSON state remains opaque and produces a warning.
5. Apply the exact hash with `vocation legacy-import-apply sha256:<hash>`.
6. Run `vocation checkpoint-create`.
7. Run `vocation checkpoint-verify` and `vocation export-audit`.
8. Keep the legacy source files until the new runtime has passed a separate recovery drill.

## Backup and Restore

Stop `vocationd` before direct store administration.

```bash
vocation store-verify
vocation store-backup ./backups/manual.vocationbak
vocation store-restore ./backups/manual.vocationbak --replace
```

Backup passphrases are masked interactive input. Restore stages and verifies the database before replacing the target. The previous target remains as a rollback image.

Restoring history older than the externally retained checkpoint digest fails closed. An intentional historical rollback additionally requires `--reanchor-checkpoint`. This verifies the checkpoint signatures and internal chain before changing the external digest. Database and digest changes roll back together if final verification fails.

An import response includes its canonical automatic rollback backup path. Restore it with explicit approval:

```bash
vocation store-rollback <canonical-runtime-backup-path> --replace --reanchor-checkpoint
```

`store-rollback` accepts only files inside the canonical runtime backup directory.

## Compatibility

`store-doctor` remains an alias for `store-verify` through v0.5. Legacy state modules remain available for validation and import compatibility, but they are no longer mutation authorities for daemon backed CLI operations.
